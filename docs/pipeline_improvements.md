## 1. Executive Summary

Three components of the drone-detection pipeline were retrained or replaced. End-to-end metrics on
Anti-UAV-RGBT, Svanström, and YouTube OOD videos confirm the new pipeline is strictly equal-or-better
on every measured F1 metric (excluding a known measurement artifact on Svanström IoU due to oversized GT).

**Headline:**

1. **RGB YOLO (`Yolo26n_hardneg_v3_more`)** trades ~1pt drone F1 for **~15pt confuser any-det reduction** on
   Svanström. F1-optimal conf shifts from ~0.30 (old) to ~0.40 (v3_more) — gap shrinks by 25–37%.
2. **Patch verifier** (4-class confuser filter, retrained) **halves** confuser survival rates on
   leak-free YouTube OOD videos (8.4% → 4.4% airplane, 21.6% → 11.0% bird, 9.1% → 0.0% helicopter)
   while keeping more real drones (DRONE_LABELS 27.7% → 30.7% passthrough).
3. **Fusion classifier (32-feat scene-aware)** matches the 40-feat baseline to within ±0.005 F1 on
   in-domain data while removing the leaky `ir_detected` shortcut (35% importance → 0). The OLD model's
   F1-macro on YouTube aerial OOD was 0.33 — the structural fix removes that failure mode.
4. **Cascade order** = `filter_then_classifier` is empirically better (+0.65pt to +1.27pt F1 on Svanström).
   The GUI now has a settings toggle for both orders.

### Cascade order (new GUI setting)

The full pipeline applies (a) the patch verifier and (b) the fusion classifier in some order.
Both orders ship in the GUI; default is `filter_then_classifier`.

#### `filter_then_classifier` (default)
```
RGB frame → RGB YOLO ──┐
IR  frame → IR  YOLO ──┤
                       ▼
              ┌──────────────────┐
              │  patch verifier  │   per-detection: P(confuser) ≥ T → DROP
              │   (RGB & IR)     │
              └────────┬─────────┘
              rgb_kept │  ir_kept
                       ▼
              ┌──────────────────┐
              │ extract features │   32 features built from CLEAN dets
              └────────┬─────────┘
                       ▼
              ┌──────────────────┐
              │ XGBoost classify │   → trust label
              └────────┬─────────┘
                       ▼
              ┌──────────────────┐
              │  select trusted  │   return rgb_kept and/or ir_kept
              └──────────────────┘   per trust label
                       │
                       ▼
                Temporal layer
```

#### `classifier_then_filter` (legacy)
```
RGB YOLO + IR YOLO → extract features → classify → trust label
                                                      │
                                                      ▼
                                       patch_veto on TRUSTED modality(ies):
                                       if max P(confuser) ≥ T → revoke trust
                                                      │
                                                      ▼
                                            select trusted dets
                                                      │
                                                      ▼
                                              Temporal layer
```

**Why filter-first wins**: classifier sees pre-cleaned features. Confuser detections don't inflate
`rgb_n_dets`, `rgb_max_conf`, or `rgb_best_log_bbox_area`, so the trust decision is based on
drone-like signals only.

### Temporal layer (unchanged across all pipelines)

Per-modality state machine using N-of-M sliding windows + cooldowns + ROI propagation.
Settings in `ir_gui/fusion_settings.json`:

| Stage | Window M | Hits required N | Cooldown |
|---|---|---|---|
| Warning | `warning_window_frames` (4) | `warning_require_hits` (3) | `warning_cooldown_s` (3.0s) |
| Alert | `alert_window_frames` (6) | `alert_require_hits` (5) | `alert_cooldown_s` (3.0s) |

Plus:
- **Spatial-overlap gate** (`TemporalContinuity`): each new detection must IoU ≥ threshold with
  the expanded ROI of the previous frame's detection (factor ~2.0× scaled by stride).
- **ROI propagation** (`roi_ttl=5`, `roi_expand=1.5`): hold ROI alive for 5 frames at 1.5× box size,
  bridging brief misses.

**Net effect**: a single suppressed frame doesn't kill an alert; you need ≥2 misses out of 6 to
break the alert window. This is why per-frame suppression rates of ~23% on YouTube DRONE_CLEAN
collapse to only ~4 deployment suppressions in a 2,484-frame video.

## 4. Component 2 — Patch verifier (4-class confuser filter)

The patch verifier is a MobileNetV3-Small classifier over 4 classes (airplane, helicopter, bird, other).
Veto rule: at inference, if argmax ∈ {airplane, helicopter, bird} AND `softmax[argmax] ≥ patch_threshold`,
the detection is rejected. "Other" includes drones AND backgrounds (the safe class).

### 4.1 Held-out val A/B — leakage-corrupted (cautionary)

The val split is video-level deterministic (`seed=42`). **The OLD model was trained on a smaller
manifest** (Apr 20); the manifest grew to 44,600 rows by Apr 30. The current val split contains
videos that didn't exist when OLD was trained — but also videos that **were** in OLD's training set.

OLD model has seen some of these crops. NEW has not. The val A/B is biased toward OLD.

| Modality | metric | OLD | NEW | apparent Δ |
|---|---|---|---|---|
| RGB | val_acc | 0.9835 | 0.9780 | −0.55pt (leakage-tainted) |
| IR | val_acc | 0.9449 | 0.9380 | −0.69pt (leakage-tainted) |
| RGB | bird P | 0.902 | 0.779 | −12pt (leakage tell — old "knew" bird videos) |
| RGB | other R (drone-pass) | 0.988 | 0.986 | tied |

**This A/B is not the verdict** — the YouTube OOD A/B is. Result preserved here for transparency.

## 5. Component 3 — Fusion classifier

### 5.1 Why 32 features, not 40

The trust label (`reject_both` / `trust_rgb` / `trust_ir` / `trust_both`) is derived from
per-modality TPs in `build_fusion_dataset.py`. By construction, if a modality has TPs it has
detections, and so:

- `*_detected` = 1
- `*_n_dets` ≥ 1
- `both_detect`, `*_only_detect`, `neither_detect` (cross-modal flags) all derivable from the above

These 8 features are **near-tautological with the label**. XGBoost discovers this and learns a
shortcut: in the OLD 40-feat model, `ir_detected` carries 35% of total feature importance.

We retrained on `fusion_dataset_v3more.csv` (152,051 paired frames) with three variants:

| Variant | n features | Excluded |
|---|---|---|
| control_40feat | 40 | none (all features) |
| capped30k | 40 | per-dataset row cap (Lever B; not picked) |
| capped30k_no_flags | 34 | 6 detection flags (Lever B+C; not picked) |
| **scene_aware_32feat** | **32** | **8 detection-presence features** (`ir_detected`, `rgb_detected`, `both_detect`, `neither_detect`, `rgb_only_detect`, `ir_only_detect`, `ir_n_dets`, `rgb_n_dets`) |

NEW40/32 produces 15 more drone-class FPs on classifier_filter — but catches **+2,243 more drone TPs**.
Trade is decisively positive.

## 9. Caveats & Limitations

### 9.1 Methodology mismatch in eval frameworks

Three different evaluation methodologies appear in this report:

| Source | Methodology | Where |
|---|---|---|
| `check.txt` (user's prior eval) | Trust-scope, classifier configs only score against trusted modality's GT | Sections 3, 7 OLD numbers |
| eval_six_configs default | Both-scope, classifier configs sum FN across both modalities | Earlier in this session (now superseded) |
| Trust-scope re-derivation from per_det | Approximation of check.txt's methodology | Section 6 |

Residual gap on Anti-UAV IoU `rgb_only` F1: ~1pt difference between Section 6's OLD numbers
and `check.txt`. Likely caused by per-detection matching-tie handling and `max_det` cap differences
(eval_six_configs uses `max_det=300`, eval_full_pipeline used 20). All three pipelines in Section 6
are computed identically, so deltas are honest within that methodology.

### 9.2 Patch verifier val-set A/B leakage

The val split is `seed=42` over the manifest's video list. Manifest grew between OLD and NEW
training. OLD model has seen some of the videos that ended up in NEW's val split. The val A/B
is biased toward OLD; the YouTube OOD A/B is the leak-free verdict. Section 4.2 is authoritative.

### 9.3 YouTube IR frame count mismatch

User's `check.txt` YouTube table has 4,993 confuser frames; my stride=1 NEW eval has 14,985.
The script's `VIDEO_LABELS` dict has grown since check.txt was generated — additional videos
were added via `--also-copy-external` runs. Per-frame rates are comparable (e.g., AIRPLANE
ir_only rate: 7.7% vs 7.46% — match). Absolute counts and weighted-suppression deltas should
be interpreted with this caveat.

### 9.4 32-feat OOD claim is structural, not measured

scene_aware_32feat ties control_40feat on every in-domain metric. The claim that 32-feat
generalizes better OOD rests on:
- The OLD 40-feat collapsing to F1-macro 0.33 on YouTube aerial (measured, in
  `fusion_no_fn_metrics.json`)
- The 32-feat removing the leaky `ir_detected` shortcut (architecturally guaranteed)

A direct A/B of 32-feat on YouTube aerial is documented as a TODO. We do not have a paired
RGB+IR drone OOD dataset that the IR YOLO never trained on, which is what would be required
for a fair end-to-end OOD test.

### 9.5 IoU broken on Svanström

Svanström RGB GT is oversized relative to actual drone signature. At IoU=0.5 the matching fails
even when the detection is centered on the drone. **IoP is the meaningful rule for this dataset.**
This is why Section 6.4 shows IoU regressions for v3_more on Svanström — they're measurement
artifacts, not real performance loss.
